# ADR-002: Choose Kafka for Message Processing with Idempotent Design

**Status:** Accepted
**Date:** 2026-03-26
**Last Updated:** 2026-03-29
**Deciders:** kfreiman

## Context

The Casino Transaction Management System requires reliable, asynchronous processing of bet and win events. Auditability and data integrity are critical; no transaction event can be lost or duplicated.

**Transaction ID Gap:** The original requirements do not define a transaction identifier. To ensure idempotency (processing the same event twice results in only one update), we must adopt a unique identifier strategy.

### Alternatives Considered for Idempotency

- **Natural Key**: Using a composite of fields (user, timestamp, amount). Fragile and complex.
- **Database-generated ID**: Simple but fails on retries because each retry generates a new ID.
- **Consumer-generated ID**: Fails on retries because the consumer might generate different IDs for the same input on different attempts.
- **Producer-generated UUIDv7 (Chosen)**: The producer assigns a time-ordered UUID at the moment of the event.

## Decision

We will use **Apache Kafka** (KRaft mode) as the message broker and **Producer-generated UUIDv7** as the mandatory idempotency key.

### Technical Stack

- **Message Broker**: Apache Kafka (KRaft mode).
- **Client Library**: `github.com/segmentio/kafka-go`.
- **Idempotency Key**: UUIDv7 (generated by Producer).
- **Delivery Semantics**: At-least-once with idempotent database writes.

## Rationale

### Why UUIDv7?

- **Time-Ordered**: Lexicographically sortable, keeping database indexes efficient and avoiding page splits.
- **Source-Generated**: Ensures the same business event always has the same identifier, even if resent by the producer.
- **Collision-Free**: Provides sufficient entropy for high-throughput systems.

### Why Kafka?

- **Replayability**: Allows re-processing messages from the log for audits or recovery.
- **Scalability**: Native support for horizontal scaling via consumer groups.
- **Durability**: Persistent log-based storage ensures messages aren't lost after consumption.

## Validation & Error Handling

### UUIDv7 Validation Rules

Consumers must validate incoming IDs before processing:

1. **Format**: Must be a valid UUID.
2. **Version**: Must be Version 7.
3. **Wait Time**: Temporal check to reject IDs with timestamps too far in the past or future (e.g., ±24h).

### Error Strategy

- **Validation Failures**: Messages with invalid IDs are moved to a Dead Letter Queue (DLQ) for manual inspection.
- **Duplicate Keys**: Handled by Database Unique Constraints; the consumer acknowledges the message but performs no action.

## Consequences

### Positive

- **Guaranteed Integrity**: Prevents double-spending/crediting via unique primary keys.
- **Performance**: UUIDv7 maintains high database insert performance.
- **Scalability**: Safe horizontal scaling of consumers.

### Negative

- **Operational Complexity**: Kafka (even without Zookeeper) requires more management than simple brokers.
- **Producer Contract**: Requires strict coordination with producer teams to include UUIDv7.
